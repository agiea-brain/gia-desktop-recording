# P0 Critical Fixes — 2026-03-26

Pre-production security and reliability fixes applied to the Gia desktop recording app.

---

## P0-1: Security — nodeIntegration + contextIsolation

**Problem:** All 3 BrowserWindows (meeting popup, onboarding, debug controls) had `nodeIntegration: true` and `contextIsolation: false`. HTML files used `require('electron')` directly. Any XSS vector would give full Node.js access — filesystem, OS commands, network.

**Fix:**
- Created dedicated preload scripts per window (`meeting-popup-preload.js`, `onboarding-preload.js`, `debug-controls-preload.js`) exposing only the IPC methods each window needs
- Set `nodeIntegration: false`, `contextIsolation: true` on all BrowserWindows
- Migrated all HTML files from `require('electron')` to `window.electronAPI` bridge
- Added `getPreloadPath()` helper for dev/packaged path resolution
- Added preload scripts to `extraResource` in `forge.config.js`
- Removed `unsafe-eval` from debug controls CSP

**Files changed:** `src/main.js`, `src/meeting-popup.html`, `src/onboarding-popup.html`, `src/debug-controls.html`, `src/debug-controls-renderer.js`, `forge.config.js`
**Files created:** `src/meeting-popup-preload.js`, `src/onboarding-preload.js`, `src/debug-controls-preload.js`

---

## P0-2: Duplicate SDK Event Listeners

**Problem:** `bootstrap()` registered `meeting-detected`, `permission-status`, `meeting-updated`, `sdk-state-change`, and `recording-ended` listeners inline (~210 lines). `ensureRecallRuntimeListenersRegistered()` registered the exact same listeners. When onboarding was already complete at startup, both paths ran — every SDK event fired its handler twice. This caused `currentMeetingInfo` to be overwritten, popups created/destroyed/recreated, and race conditions in recording state.

**Fix:** Replaced ~210 lines of inline listener registration in `bootstrap()` with a single call to `ensureRecallRuntimeListenersRegistered()`, which already has a guard flag (`recallRuntimeListenersRegistered`) preventing double-registration.

**Files changed:** `src/main.js`

---

## P0-3: 11 Dead Preload Methods

**Problem:** `preload.js` exposed 11 methods via `window.electronAPI` that had no corresponding `ipcMain.handle()` in `main.js`: `saveMeetingsData`, `loadMeetingsData`, `deleteMeeting`, `generateMeetingSummary`, `generateMeetingSummaryStreaming`, `startManualRecording`, `stopManualRecording`, `debugGetHandlers`, `checkForDetectedMeeting`, `joinDetectedMeeting`, `getActiveRecordingId`. Any call to these would return a Promise that never resolves — silently hanging the caller.

**Fix:** Removed all 11 dead methods from `preload.js`.

**Files changed:** `src/preload.js`

---

## P0-4: Auth Expiration During Recording — No Recovery

**Problem:** When `api.onAuthExpired` fired (401 response), the handler immediately showed a re-login popup. During active recording this meant: recording continued with a dead token, subsequent API calls (meeting URL registration, diagnostics) all failed, and no attempt was made to silently refresh.

**Fix:** `api.onAuthExpired` now first attempts silent token refresh via `getStoredAccessToken({ allowRefresh: true })`. If the refresh token is still valid, the auth token is restored transparently — no user interruption. Only if silent refresh fails does it clear the token and show the re-login popup.

**Files changed:** `src/main.js`

---

## P0-5: No Fetch Timeout in API Client

**Problem:** All 4 `fetch()` calls in `api.js` (`getUploadToken`, `registerMeetingUrl`, `getUserProfile`, `updateDesktopSdkDiagnostics`) had no timeout. A hung or slow backend would block the app indefinitely — e.g., user clicks "Start Recording" and the upload token request hangs forever with no feedback.

**Fix:** Added `_fetch()` helper method using `AbortController` with 60-second timeout. All 4 fetch calls now route through `this._fetch()`. On timeout, the request aborts and throws, allowing callers to handle the error (show dialog, retry, etc.).

**Files changed:** `src/utils/api.js`
